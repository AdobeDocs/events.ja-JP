# エキスパートへの質問 — Velocity の測定

を使用して速度を測定および追跡する方法を説明します。 [!DNL Workfront] レポート。 本ワークショップは 2019 年 8 月 14 日に開催されました。

>[!VIDEO](https://video.tv.adobe.com/v/341057/?quality=12)

## プレゼンテーションで使用するカスタムフィールド

以下の計算をコピー&amp;ペーストして、時間を節約します。

**最初のコミット日**

形式：日付

計算：

```
IF(ISBLANK({DE:First Commit Date}),{defaultBaseline}.{plannedCompletionDate},{DE:First Commit Date})
```

**最初の期間**

形式：テキスト

計算：

```
IF(ISBLANK({DE:First Duration}),{defaultBaseline}.{durationMinutes},{DE:First Duration})
```

**WTC 比率**

形式：数値

計算：

```
ROUND(DIV({actualDurationMinutes},{DE:First Duration}),1)
```

**WTO（コミット間）比率ステータス**

形式：テキスト

計算：

```
IF({DE:Work-to-Commit Ratio}>2,"Terrible",IF({DE:Work-to-Commit Ratio}>1.6,"Poor",IF({DE:Work-to-Commit Ratio}>1.2,"Not Bad","Excellent")))
```

**調整された速度**

形式：数値

計算：

```
ROUND(DIV({actualDurationMinutes},{durationMinutes}),1)
```

**調整された速度のステータス**

形式：テキスト

計算：

```
IF({DE:Adjusted Velocity}>2,"Terrible",IF({DE:Adjusted Velocity}>1.6,"Poor",IF({DE:Adjusted Velocity}>1.2,"Not Bad","Excellent")))
```

## Q&amp;A

**質問**

こんにちは、このウェビナーを整理していただき、ありがとうございます。 Workfrontのフィールドに関して質問があります。 このシステムでは、「状態」という、ステータスと条件を組み合わせたカスタムフィールドを作成しました。 この「状態」フィールドには、Tableau のデータ抽出にとって非常に重要な、千のプロジェクトの多くの彫像が含まれています。 ただし、次に、このフィールドを削除し、代わりにネイティブフィールドの「条件」フィールドを使用します。 データを失わずにこのフィールドを反転させる方法をご存知ですか。 今すぐデータを失わずにそれを行うことを考えられるのは、プロジェクトからプロジェクトに手動で切り替えることだけです。

**回答**

このような状況では、フィルタリングと一括編集を使用して、State カスタムフィールドに基づいて「条件」フィールドにデータを入力する作業を半自動化できます。

手順は次のとおりです。

1. 条件値にマッピングする状態値を決定します。 例えば、「State」の値が「Late」と「Very Late」で、両方とも「In Troul」の条件値にマッピングされているとします。
1. 状態値が「遅延」および「非常に遅い」のすべてのプロジェクトを示すプロジェクトレポートを作成する
1. レポートを実行します。 すべてのプロジェクトを表示していることを確認します（レポートの右下にあるオプションを参照）。
1. 列見出しの付いたバーのレポートの左上にあるチェックボックスをクリックします。 これにより、レポート内のすべてのプロジェクトが選択されます
1. レポートリストの上の「編集」ボタンをクリックします。
1. 「条件タイプ」を「手動」に設定します。
1. 「条件」フィールドを「問題あり」に設定します。
1. 「変更を保存」をクリックします。


**質問**

優れた、悪くないなどの定義について教えてください。

**回答**

これはほんの一例ですが、次に設定方法を示します。 最初に、2 つのインデックスを計算しました。

調整された速度

この数式は「実績期間/計画期間」（プロジェクトの「期間」フィールドに格納されます）です。 プロジェクトの「計画期間」は、プロジェクトが再計画されるたびに変更されるので、「計画期間」は最終的な再計画を表します。

WTC 比率

この式は、調整された速度に似ていますが、最後の再計画の「計画期間」の値を使用する代わりに、顧客に最初に約束した計画期間を使用する場合を除きます。 元のベースラインにはこの情報が含まれていると想定しています（今後、正確なデータを取得できるように、この方法でプロジェクトを計画するようプロジェクトマネージャーに依頼する予定です）。 この期間の値は元のベースラインから取り込み、「最初の期間」と呼びます。

実際の期間を計画期間または最初の期間で割ることで、目標に到達した時間を示す数値が得られます。 「計画期間」または「初期期間」が「実際の期間」と等しい場合、インデックスは 1 になります。 実際の期間が長い場合、回答は 1 より大きくなります。 デートの日に会った人の数が多いほど悪くなる。

そこで、調整された速度と WTC 比の両方に対して、次のようなステータスを割り当てることにしました。

* 1.1 以下では、Excelrent と呼んでいます。
* 1.2 から 1.5 私は悪くないと呼びました。
* 1.6 から 1.9 私は貧しいと呼びました。
* 2 つ以上の私は恐ろしいと呼びました

**質問**

プロジェクトの実行に要する時間を追跡するには、ワーカーは何をおこなう必要がありますか？

**回答**

ここでは、プロジェクトに費やした実際の時間は追跡せず、単に期間を追跡して比較するだけです。 しかし、時間を追跡し、計画時間に対する実際の時間を使用して速度を計算する場合は、計画時間と実際の時間を比較することで、同じタイプのレポートを実行できます。 元のベースラインからも計画時間を取り込む必要があります。

**質問**

Velocity に使用するフィルターを指定できますか？

**回答**

Velocity レポートに 2 つのフィルタールールを使用しました。

* カテゴリ/ ID /次と等しい/ WPI プロジェクトデータ ( これは、すべての計算フィールドを含むカスタムフォームです。 このカスタムフォームを使用しているプロジェクトのみを表示したい
* プロジェクト//ステータス/次と等しい/> 完了 ( 完了したプロジェクトは、すべての処理に要した時間を表す実際の期間値を持つので、完了したプロジェクトのみを表示します。 現在のプロジェクトでは、速度を計算するための正確な実際の期間が提供されません )

また、Web セミナーで管理できる小さいレポートを維持するための他のフィルターも追加しましたが、実稼働環境では、特定の期間に WPI カスタムフォームを持つすべてのプロジェクトを表示したいと考えられます。 そのプロジェクトの実績完了日をフィルタリングできます。

**質問**

プロジェクトをコピーする場合、新しいプロジェクトに同じベースラインが適用されますか。

**回答**

いいえ、基準はコピーされたプロジェクトに含まれません

**質問**

タスクの承認プロセスで、プロジェクト内のタスクごとに監査証跡を提供するレポートを、各承認者の日時スタンプ付きで作成する方法を教えてください。

**回答**

タスクレポートを作成します。 「列（表示） 」タブで、「列の追加」をクリックします。 「次の列に表示：」ボックスに「approv」と入力します。 これにより、レポートで使用できる様々な承認フィールドが表示されます。 最初は（ID を除く）すべての項目に列を追加し、表示される情報を確認できるようにすることをお勧めします。

次に「フィルター」タブに移動し、次のフィルタールールを追加します。

タスク >> 承認 >> 次と等しい >> True です。 承認が添付されたタスクのみが表示されます。

必要に応じて、他のフィルターを追加します。

**質問**

配達確認レポートを作成します。 プロジェクトのリストで、配達確認の数と、その配達確認のバージョンの数を示します。

**回答**

ドキュメントレポートを作成します。

デフォルトのビューには、バージョン番号が表示されます。 そのままにしておきたいのですが、他の列は変更できます。

プロジェクト名別にレポートをグループ化します。

現在のバージョンでレポートをフィルター：配達確認 ID が空ではありません。

これにより、各プロジェクトのすべての配達確認のリストが表示されます。 配達確認ごとに行が作成され、バージョン番号が表示されます（バージョンの合計数と同じ）。

**質問**

Velocity をタスク・レベルで使用できますか？ プロジェクトレベルではなく

**回答**

はい。ただし、プロジェクトのカスタムフォームをコピーし、そのフォームからタスクのカスタムフォームを作成する必要があります。 次に、「最初のコミット日」フィールドで計算を編集し、参照を「デフォルトの基準計画」から「デフォルトの基準タスク」に変更する必要があります。 最初の期間も同じようにします。 その後、測定する任意のタスクにタスクのカスタムフォームを添付できます。 これらのレポートに対して、プロジェクトレポートではなくタスクレポートを作成する必要があります。 ただし、元のプロジェクトのベースラインがデフォルトのベースラインとして設定されていることを確認する必要があります。 すべてのタスクデータは、プロジェクトデータと同じベースラインに保持されます。

**質問**

最初のコミット日は、プロジェクト所有者が手動で設定する必要がありますか？ または、既存のフィールドから取り込むことはできますか？

**回答**

最初のコミット日は、デフォルトのベースラインから取り込まれます。 デフォルトの基準線が元の基準線である限り、これは、最初に [ 現在 ] ステータスに設定された時点のプロジェクトの計画完了日を表示します。

**質問**

カスタムフォームの計算フィールドは、定期的に更新する必要があります。正しいですか？ それとも、今、それは自動的に一晩 ( または他のトリガーで ) 起こりますか？

**回答**

計算フィールドは再計算されます。

* ユーザーがオブジェクトを編集したとき
* アクティブ化された一括編集時にカスタム式を再計算
* 選択した「以前の計算を更新」オプションを使用したフォームへの変更

**質問**

速度で「期間」を考慮する場合、「プロジェクトプリファレンスでの完了率」は「期間」に基づく必要がありますか？

**回答**

いいえ。「プロジェクトの環境設定」オプションでは、「完了率」の計算方法のみが参照されます。 デュレーションの値自体は、この設定の影響を受けません。

**質問**

最初と計画期間の違いは何ですか？

**回答**

最初の期間は、プロジェクトに要する顧客に最初に約束した日数です。 この数は、プロジェクトが Planning から Current に変更されたときに記録された元の基準線から取得されます。

「計画期間」は、プロジェクトの開始から計画完了日までの日数です。 この 2 つの期間は最初は同じですが、プロジェクトが再計画され、完了予定日が変更された場合は、計画期間も変更されました。

Velocity レポートの値は、計画期間が最初の期間からどの程度変化したかを確認できることから得られます。 これは、最初にベースラインの記録を開始した時点まで、Planning から Current に変更されたプロジェクトとして遡ることができます。

**質問**

すべてのレポートで同じになるように、ワーカーを色別に設定できますか？

**回答**

タスクレポートで「割り当て先/名前」でグループ化した場合は、「グラフ」タブで特定の作業者に色を割り当てることができます。 「グラフ」タブの「割り当て先/> 名前」ボックスの横にある「カスタムカラー」オプションを選択し、各作業者の色を追加します。

**質問**

タスクレベルのカスタムフォームを調べる 1 つの領域を持つダッシュボードを作成して、その領域が存在するかどうかを確認し、特定のフィールドが空白でない場合は 2 番目の領域を作成できるかどうかを判断しようとしています。 これは可能ですか？

**回答**

私があなたの質問を理解しているかどうか見てみましょう。 「Tammy Form」という名前のタスクのカスタムフォームがあり、その中に「Tammy Field」という名前のフィールドがあるとします。

Tammy フォームが添付され、Tammy フィールドに値が設定されているすべてのタスクを表示するタスクレポートが必要です。

はい、できます。 2 つのフィルタールールを持つフィルターがタスクレポートに必要になるだけです。

カテゴリ >> ID がタミーと等しいフォーム

タスク >> タミーフィールドが空ではありません

**質問**

ドキュメントライブラリ内の特定の名前の付いたドキュメントを検索するレポートを作成する方法はありますか。 特定のドキュメントが存在するかどうかを測定するダッシュボードの一部。

**回答**

はい。ドキュメントレポートを作成する必要があります。 レポートを実行するたびに、特定のドキュメント名を指定する必要があるようです。 その場合は、「レポート・オプション」に移動し、「レポート・プロンプト」を選択することをお勧めします。 [ ドキュメント >> 名前 ] のプロンプトを追加します。

**質問**

グラフタブに表示されない色/16 進数の値を選択できますが、新しい色です。新しい色は 16 進数の値です。例えば、ブランドの色から新しい色を選択して、レポートをカスタマイズできますか？

**回答**

はい、検索可能な任意のRGBコードを入力できます。 これは、カラーに含まれる赤、緑、青の量を示す標準コードです。 Workfrontは、000000から FFFFFF までの 16 進数値を受け取るので、ブランドカラーのコードがわかっていれば、それを使用できます。

**質問**

ダッシュボードでレポートの定義を再度述べる（各レポートの測定の文を提供する）ことはできますか？

**回答**

調整済み速度ステータスグラフ

> これは、プロジェクトの実績期間と計画期間を比較する際に、プロジェクトを時間通りに完了した場合にどの程度良く行ったかを示します。 計画期間は、プロジェクトがライフサイクル中に再計画された際に調整されました。

WTO-COMMIT 比率ステータス・グラフ

> これは、プロジェクトの実際の期間と最初の期間を比較する際に、プロジェクトを時間通りに完了した場合にどの程度良く行ったかを示します。 最初の期間は、プロジェクトの完了に要する時間をお客様に約束した最初の時間です。 最初の期間は、プロジェクトが最初に「計画」ステータスから「現在」ステータスに変更された時点の期間値から計算されます。 これは、元のベースラインで記録された期間です。

速度ステータスリストレポート

> このレポートには、グラフ内の同じプロジェクトに関するすべての計算済みカスタムフィールドと有効期限が含まれます。 その目的は、計算を確認し、必要に応じて詳細情報を得ることを可能にすることです。

**質問**

X 軸に新しいデータを追加した方法を教えてください。

**回答**

レポート内の任意の項目でグループ化する場合、グラフを作成できます。 次に、「グラフ」タブで X 軸または Y 軸を設定できます。

**質問**

最初の期間と実際の期間の違いを調べてみてください。

**回答**

最初の期間は、プロジェクトに要する顧客に最初に約束した日数です。 この数は、プロジェクトが Planning から Current に変更されたときに記録された元の基準線から取得されます。

実際の期間は、プロジェクトの開始から実際の完了日までの日数です。

**質問**

このレポートに対するプロジェクトの基準要素はどのように表示されますか？

**回答**

プロジェクトの元の基準計画には、プロジェクトが最初に「現在」のステータスに変更された時点で存在した計画完了日と計画期間が含まれます。 プロセスでプロジェクトを「現在」に設定する前に計画する場合は、これはプロジェクトの完了にコミットした日付を表します。

**質問**

古いプロジェクトに新しいフォームを一括で適用する方法はありますか？

**回答**

はい、リストから複数のプロジェクトを選択できます。 これを行うと、リストの左上に「編集」オプションが表示されます。 複数のオブジェクトが選択されている場合に「編集」をクリックすると、「一括編集」と呼ばれる操作になります。 この方法で、複数のプロジェクトにカスタムフォームを追加できます。

多数のプロジェクトにカスタムフォームを追加するためのショートカットは、フィルターを適用して、必要なプロジェクトのみを含めるレポートを作成することです。 次に、プロジェクトを個別に選択する代わりに、リスト内の最初のチェックボックスの上にあるチェックボックスをクリックして、リスト全体を選択します。

**質問**

割り当てレポートのグループ内で重複エントリを削除できますが、グループ間では削除できませんか？

**回答**

リストレポートでのグループ化を考える最善の方法は次のとおりです。

まず、「フィルター」タブを使用して、リストに表示する項目を制御します。 重複するエントリはありません。 フィルターが各オブジェクトに適用されます。 フィルターを通過すると、リストに 1 回表示されます。表示されない場合は、一切表示されません。

次のグループ化は、フィルターされたリストに適用されます。 グループ化は、リスト内のオブジェクトに関する 1 つのものを識別します。例えば、リスト内のオブジェクトのポートフォリオの名前などです（リスト上でグループ化することはできません。1 つのものに対してのみ）。 次に、同じポートフォリオ内のすべてのプロジェクトと同様に、同じ値を持つすべてのオブジェクトがそのグループに表示されます。 ポートフォリオが選択されていないプロジェクトは、「値なし」という名前のグループに表示されます。

その結果、複数のグループ化で表示されるオブジェクトはありません。 また、リストにオブジェクトが表示されるかどうかは、フィルターによって完全に制御されます（また、レポートを実行するユーザーに表示する権限がある場合）。

**質問**

Velocity を追跡する他のレポートをお勧めしますか？ 詳細を調べるのに十分な時間がないことを知っているので、高いレベルのレコメンデーションは素晴らしいです。

**回答**

他のレポートと同様に、まず何を知りたいかを決める必要があります。 次の手順では、その情報にアクセスします。場合によっては、情報の追跡を開始する必要があります。

実際の期間と計画期間の 2 種類を比較することにした理由の 1 つは、速度に関する興味深い洞察を提供したと思ったからです。 データも既に使用可能なので、追跡を開始する必要はありませんでした。 数回の計算を行うと、データを抽出したフォームを抽出してレポートできます。

ただし、レポート対象のタスクやプロジェクトに関する他の情報を追跡する場合も同じです。

Workfrontには velocity レポートが組み込まれていないので、velocity を決定し、何を追跡する必要があるかを知りたいと思うときに、あなたとチームがブレインストーミングすることをお勧めします。

**質問**

COLUMN レベルでは何も計算できますか。 カスタムフォームから計算フィールドを呼び出す代わりに、

**質問**

時間の入力が発生した時間と実際に実行された時間を、レポートで（どのフィールドで）分離する方法を教えてください。

**回答**

Hour オブジェクトの「 Entry Date 」フィールドは、作業時間の日付を示します。 これにより、入力日が他のオブジェクトとは異なります。ここでは、オブジェクトが作成された日付を意味します。 時間に作成日がない場合でも、「最終更新日」があります。この「最終更新日」は最初に作成日で、その後時間が編集された日です。

**質問**

現在デフォルトとして設定されている基準線に対して表示するフィールドがレポートに表示されるので、基準線を変更し、レポートを更新して新しい基準線を含むフィールドを表示できます。

しかし、タスクに関する情報を図で確認したい場合は、ガントチャート機能を使用して実行できます。 （[ 自動保存 ] の右上にあるガントアイコンを使用して）タスクリストで [ ガント ] をオンにし、右下の [ 設定 ] アイコンに移動して、それをクリックします。 「ベースライン」ボックスをオンにすると、すべてのベースラインが表示されます。 一度に 1 つずつ選択し、タスクの変更をガントビューで確認できます。

**質問**

レポートを作成して、先月など、定義した期間のステータスの変化を調べる方法。

**回答**

Workfrontの Analytics 機能を使用すると、ステータスの変更などの履歴データをすばやく表示できます。

ただし、メモレポートを使用して、ステータス変更情報を取得することもできます。 「プロジェクトステータス」フィールドを追跡している場合は、フィルターを適用して、プロジェクトのステータスの変更を表示できます。

最初に、セットアップ/インターフェイス/フィードを更新に移動し、プロジェクトステータスが追跡中の組み込みフィールドの 1 つであることを確認します。 追加する必要がない場合は、

メモレポートを作成し、次の操作を行います。

「列（表示） 」タブで、次の操作をおこないます。

* 「監査テキスト」の「メモテキスト」列を置き換えます。 ステータスの変更元と変更先に関する情報が表示されます
* 「Project: Name」列と「Entry Date」列はそのままにします。
* 「入口日」列をクリックし、列設定パネルで「この列で並べ替え」をオンにします。 上位の並べ替えで最新のステータスの変化を降順に表示する場合。

「グループ化」タブで、以下の操作をおこないます。

* プロジェクト別にグループ化：名前

「フィルター」タブで、次のフィルタールールを作成します。

* 注意 >> 監査タイプ >> 次と等しい >> ステータスの変更
* 「メモの入力日」でフィルタリングする他のルールを追加します。 これをフィルターから除外し、代わりにレポートプロンプトを使用するとよいでしょう。
* 必要に応じて、プロジェクト、ポートフォリオ、その他のデータをフィルタリングします。
