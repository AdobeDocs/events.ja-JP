# エキスパートに問い合わせる – 速度の測定

[!DNL Workfront] レポートを使用して速度を測定および追跡する方法を説明します。 このワークショップは2019年8月14日に録音されました。

>[!VIDEO](https://video.tv.adobe.com/v/341057/?quality=12)

## プレゼンテーションで使用されるカスタムフィールド

以下の計算をコピー&amp;ペーストして、時間を節約します。

>[!NOTE]
>
>カスタムフィールド計算の構文は、2019年のプレゼンテーション以降に変更されましたが、プレゼンテーションで提示された概念やその他の指示は依然として正確です。
>**以下に含まれる計算は、最新の構文ルールを反映するように更新されました。**

**最初のコミット日**

形式：日付

計算：

```
IF(ISBLANK({DE:First Commit Date}),{defaultBaseline}.{plannedCompletionDate},{DE:First Commit Date})
```

**最初の期間**

フォーマット：テキスト

計算：

```
IF(ISBLANK({DE:First Duration}),{defaultBaseline}.{durationMinutes},{DE:First Duration})
```

**作業対確約率**

形式:Number

計算：

```
ROUND(DIV({actualDurationMinutes},{DE:First Duration}),1)
```

**作業対コミット比率の状態**

形式:Text

計算：

```
IF({DE:Work-to-Commit Ratio}>2,"Terrible",IF({DE:Work-to-Commit Ratio}>1.6,"Poor",IF({DE:Work-to-Commit Ratio}>1.2,"Not Bad","Excellent")))
```

**調整済み速度**

形式:Number

計算：

```
ROUND(DIV({actualDurationMinutes},{durationMinutes}),1)
```

**調整済み速度の状態**

形式:Text

計算：

```
IF({DE:Adjusted Velocity}>2,"Terrible",IF({DE:Adjusted Velocity}>1.6,"Poor",IF({DE:Adjusted Velocity}>1.2,"Not Bad","Excellent")))
```

## Q&amp;A

**質問**

ウェビナーを開催いただきありがとうございます。 WorkfrontのFieldについて質問します。 システムでは、ステータスと条件を組み合わせた「State」というカスタムフィールドを作成しました。 この状態フィールドには、Tableau データ抽出で非常に重要な数千のプロジェクトの多くの像が含まれています。 ただし、このフィールドを削除し、代わりにネイティブフィールドであるConditon フィールドを使用します。 データを失うことなくこのフィールドを反転する方法を知っていますか。 今すぐデータを失うことなく、手作業でプロジェクトからプロジェクトに切り替えることしかできません。

**回答**

このような状況では、フィルタリングと一括編集を使用して、状態カスタムフィールドに基づいて条件フィールドに入力する手間を半自動化できます。

次の手順を実行します。

1. 条件の値にマッピングする状態の値を決定します。 例えば、Stateの値が「Late」で「Very Late」で、両方ともConditionの値が「In Trouble」にマッピングされているとします
1. State値が「Late」および「Very Late」のすべてのプロジェクトを示すプロジェクトレポートを作成します
1. レポートを実行します。 すべてのプロジェクトが表示されていることを確認します（レポートの右下にあるオプションを参照）
1. 列の見出しを含むバーのレポートの左上にあるチェックボックスをクリックします。 レポート内のすべてのプロジェクトが選択されます
1. レポートリストの上にある「編集」ボタンをクリックします
1. 条件タイプを手動に設定
1. 「条件」フィールドを「問題あり」に設定します
1. 「Save Changes」をクリックします


**質問**

「優れている」、「悪くない」などの定義は？

**回答**

例をいくつか紹介しますが、次のようにして設定します。 まず、2つのインデックスを計算しました。

調整済み速度

この式は、「実際の期間/予定期間」（プロジェクトの「期間」フィールドに格納されます）です。 プロジェクトの予定期間は、プロジェクトが再計画されるたびに変更される可能性があるため、予定期間は最終的な再計画を表します。

作業対約束比率

この式は調整済み速度に似ていますが、最終的な再計画の予定期間の値を使用する代わりに、最初にお客様に約束した予定期間を使用する必要があります。 私たちは、元のベースラインにこの情報が含まれていると仮定しています（そして、正確なデータを取得できるように、プロジェクトマネージャーにこのようにプロジェクトを計画するように依頼することを今後は計画しています）。 このデュレーション値を元のベースラインから取得し、「最初のデュレーション」と呼びました。

「実際の期間」を「予定の期間」または「最初の期間」で割ることで、目標にどれくらい近づいたかを知ることができる数字になります。 予定期間または最初の期間が実際の期間と等しい場合、インデックスは1に等しくなります。 実際の期間が大きい場合、答えは1より大きくなります。 デートの相手にぼくが悪いことをした数が多ければ多いほど。

そのため、調整ベロシティと作業対コミット比率の両方に、次のようにステータスを割り当てることにしました。

* 1.1以下は素晴らしい名前です。
* 1.2から1.5まで、私は悪くないと呼びました。
* 1.6から1.9まで私は貧乏と呼びました。
* 2つ以上の私は恐ろしいと呼ばれていました。

**質問**

プロジェクトに要する時間を追跡するために、従業員は何をする必要がありますか？

**回答**

実際にプロジェクトに費やした時間を追跡するのではなく、期間を追跡および比較するだけです。 しかし、時間を追跡しており、実際の時間を予定時間よりも多く使用して速度を計算する場合は、予定時間を実際の時間と比較することで、同じタイプのレポートを作成できます。 元のベースラインから予定時間も取得する必要があります。

**質問**

Velocityに使用するフィルターを指定できますか？

**回答**

速度レポートには、次の2つのフィルタールールを使用しました。

* カテゴリ >> ID >>次と等しい>>> WPI プロジェクトデータ（これは、すべての計算フィールドを含むカスタムフォームです。 このカスタムフォームを使用しているプロジェクトだけを表示したい）
* プロジェクト/ステータス/次と等しい/完了（完了したプロジェクトは、完了したプロジェクトを表示する場合にのみ表示されます。完了したプロジェクトには、完了に要した時間を表す「実際の期間」の値が表示されます。 現在のプロジェクトでは、速度を計算するための正確な実際の期間は提供されません）

また、ウェビナーで管理できる程度にレポートを小さく抑えるために、他のフィルタリングも追加しました。ただし、本番環境では、特定の期間にWPI カスタムフォームを使用するすべてのプロジェクトを確認する必要があります。 その場合は、プロジェクトの実際の完了日でフィルタリングできます。

**質問**

プロジェクトをコピーする場合、新しいプロジェクトにも同じベースラインが適用されますか？

**回答**

いいえ、ベースラインはコピーされたプロジェクトには含まれません

**質問**

タスク承認プロセスの場合、各承認者の日時スタンプを含むプロジェクトのタスク別の監査証跡を提供するレポートを作成する方法を教えてください。

**回答**

タスクレポートを作成します。 「列（表示）」タブで、「列を追加」をクリックします。 「この列に表示：」ボックスに「approv」と入力します。 これにより、レポートで利用できるさまざまな承認フィールドが表示されます。 最初に（任意のIDを除く）すべての列を追加することをお勧めします。その後、表示される情報を確認できます。

次に、「フィルター」タブに移動し、次のフィルタールールを追加します。

タスク >>は承認>>次と等しい>> Trueです。 承認が添付されたタスクのみが表示されます。

必要に応じてフィルターを追加します。

**質問**

プルーフレポートを作成する。 プルーフの数とそのプルーフのバージョン数を示すプロジェクトのリスト。

**回答**

ドキュメントレポートを作成します。

デフォルトのビューには、バージョン番号が表示されます。 そのフィールドは残しますが、他の列を変更することもできます。

プロジェクト名でレポートをグループ化します。

現在のバージョン :Proof IDでレポートをフィルター処理すると、空白になっていません。

これにより、各プロジェクトのすべてのプルーフのリストが表示されます。 各プルーフの行があり、バージョン番号が表示されます（これはバージョンの合計数と同じです）。

**質問**

タスクレベルでベロシティを利用できますか？ 把握できるようになりました？

**回答**

はい。ただし、プロジェクトのカスタムフォームをコピーし、そこからタスクカスタムフォームを作成する必要があります。 次に、「最初のコミット日」フィールドで計算を編集し、参照を「デフォルトのベースライン」に「デフォルトのベースラインタスク」に変更する必要があります。 同じことを最初の期間に行います。 その後、タスクカスタムフォームを測定するタスクに添付できます。 これらのタスクのプロジェクトレポートではなく、タスクレポートを作成する必要があります。 ただし、元のプロジェクトベースラインがデフォルトのベースラインとして設定されていることを確認する必要があります。 あらゆるタスクデータは、プロジェクトデータと同じベースラインに保たれます。

**質問**

最初のコミット日は、プロジェクト所有者が手動で設定する必要がありますか？ それとも既存のフィールドから取り込むことができますか？

**回答**

最初のコミット日は、デフォルトのベースラインから取得されます。 デフォルトのベースラインが元のベースラインである限り、プロジェクトが最初に現在のステータスに設定された時点での予定完了日が表示されます。

**質問**

カスタムフォームの計算フィールドを定期的に更新する必要がありますか？ あるいは、今すぐ自動的に一晩（または他のトリガーで）起こるのでしょうか？

**回答**

計算フィールドが再計算されます。

* ユーザーがオブジェクトを編集すると
* アクティブ化された再計算カスタム式を使用した一括編集時
* 「以前の計算を更新」オプションを選択したフォームの変更

**質問**

速度が期間を考慮している場合、「プロジェクトの環境設定」の「完了率」は期間に基づくべきですか？

**回答**

いいえ、プロジェクト環境設定オプションは、完了率の計算方法のみを指します。 デュレーション値自体は、この設定の影響を受けません。

**質問**

1回目とプラン期間の違いは何ですか？

**回答**

「最初の期間」とは、プロジェクトが最初に顧客に約束した日数です。 この数値は、プロジェクトが計画から現在に変更されたときに記録された元のベースラインから取得されます。

予定期間とは、プロジェクトの開始から予定完了日までの日数です。 当初、これら2つの期間は同じですが、プロジェクトが再計画され、予定完了日が変更された場合、予定期間も変更されました。

速度レポートの価値は、計画期間が最初の期間からどれだけ変化したかを確認できるようになったことから得られます。 これは、プロジェクトが「計画」から「現在」に変更されたときに、ベースラインを最初に記録し始めた時期にまでさかのぼります。

**質問**

すべてのレポートで同じになるようにカラーでワーカーを設定できますか？

**回答**

タスクレポートで「割り当て先/名前」でグループ化する場合、「グラフ」タブで特定のワーカーにカラーを割り当てることができます。 「チャート」タブの「割り当て先/名前」ボックスの横にある「カスタム色」オプションを選択し、各作業者の色を追加するだけです。

**質問**

タスクレベルのカスタムフォームを参照する1つの領域を持つダッシュボードを作成して、それが存在するかどうかを確認し、特定のフィールドが空白でない場合はセカンダリを作成できるかどうかを判断しようとしています。 これは可能ですか？

**回答**

ご質問の内容を理解したうえで。 タミーフィールドというフィールドを持つタミーフォームというタスクカスタムフォームがあるとします。

タミーフォームが添付されているすべてのタスクと、タミーフィールドが値を持つ場所を表示するタスクレポートが必要です。

はい、その通りです。 タスクレポートに2つのフィルタールールを含むフィルターが必要になります。

カテゴリ >> IDがタミーフォームと等しい

タスク >> タミーフィールドが空白ではない

**質問**

ドキュメント ライブラリで特定の名前付きドキュメントを検索するレポートを作成する方法はありますか？ ダッシュボードの一部で、特定のドキュメントが存在するかどうかを測定します。

**回答**

はい。 ドキュメントレポートを作成する必要があります。 レポートを実行するたびに、特定の文書名を指定するように聞こえます。 そのような場合は、「レポートオプション」に移動し、「レポートプロンプト」を選択することをお勧めします。 文書/名前のプロンプトを追加します。

**質問**

「グラフ」タブに表示されていないカラー/16進数の値を選択できますが、それは新しいカラーで、例えばブランドカラーから新しいカラーを選択してレポートをカスタマイズできるようにするには、新しい16進数の値です。

**回答**

はい、見つけることができた可能性のあるRGB コードを入力できます。 これは、カラーに含まれる赤、緑、青の量を示す標準コードです。 Workfrontでは、000000からFFFFFFまでの任意の16進数の値を受け入れるので、ブランドカラーのコードを知っていれば使用できます。

**質問**

ダッシュボードにレポートの定義を再度記載してください（各レポートの測定値を記載してください）。

**回答**

調整済み速度ステータスチャート

> これは、プロジェクトの実際の期間と予定期間を比較する際に、プロジェクトを予定どおりに完了した際の成功度を示します。 プロジェクトがライフサイクル中に再計画されたため、予定期間が調整されました。

作業対約定比率のステータス チャート

> これは、プロジェクトの実際の期間と最初の期間を比較する際に、プロジェクトを予定どおりに完了した際にどの程度成果を上げたかを示しています。 「最初の期間」とは、プロジェクトの完了までにかかる時間をお客様に約束した最初の時間です。 最初の期間は、計画から現在のステータスに最初に変更されたときのプロジェクトの期間値から計算されました。 これは、元のベースラインに記録された期間です。

速度ステータスリストレポート

> このレポートには、グラフ内の同じプロジェクトのすべての計算されたカスタムフィールドと重要な日付が含まれています。 その目的は、計算を確認し、必要に応じてより多くの詳細情報を入手できるようにすることです。

**質問**

新しいデータをx軸に追加する方法を教えてください。

**回答**

レポート内の項目をグループ化すると、グラフを作成できるようになります。 その後、「グラフ」タブでX軸またはY軸を設定できます。

**質問**

最初のデュレーションと実際のデュレーションの違いを説明できますか？

**回答**

「最初の期間」とは、プロジェクトが最初に顧客に約束した日数です。 この数値は、プロジェクトが計画から現在に変更されたときに記録された元のベースラインから取得されます。

実際の期間は、プロジェクトの開始から実際の完了日までの日数です。

**質問**

プロジェクトのベースラインは、このレポートにどのように反映されますか？

**回答**

プロジェクトの元のベースラインには、プロジェクトが最初に現在のステータスに変更されたときに存在した予定完了日と予定期間が含まれます。 現在に設定する前にプロセスがプロジェクトを計画する場合は、プロジェクトを完了する日付を表します。

**質問**

新しいフォームを古いプロジェクトに一括適用する方法はありますか？

**回答**

はい、リストから複数のプロジェクトを選択できます。 これを行うと、リストの左上に「編集」オプションが表示されます。 複数のオブジェクトが選択されている場合に「編集」をクリックすると、「一括編集」と呼ばれる部分に移動します。 この方法で、複数のプロジェクトにカスタムフォームを追加できます。

多数のプロジェクトにカスタムフォームを追加するためのショートカットは、フィルターを使用してレポートを作成し、必要なプロジェクトのみを含めるようにすることです。 次に、プロジェクトを個別に選択する代わりに、リストの最初のチェックボックスの上にあるチェックボックスをクリックするだけで、リスト全体を選択できます。

**質問**

割り当てレポートのグループ内から重複するエントリを削除できますが、グループ間で削除することはできません。

**回答**

リストレポートのグループ化について考える最良の方法は次のとおりです。

まず、「フィルター」タブを使用して、リストに表示される項目を制御します。 重複したエントリはありません。 フィルターは各オブジェクトに適用されます。 フィルターを通過した場合、リストに1回表示され、そうでない場合はまったく表示されません。

次のグループ化は、フィルター処理されたリストに適用されます。 グループ化は、リスト内のオブジェクトに関して、ポートフォリオの名前など、1つのものを識別します（リスト上ではグループ化できず、1つのものだけにグループ化します）。 同じ値を持つすべてのオブジェクトが、同じポートフォリオ内のすべてのプロジェクトと同様に、そのグループに表示されます。 ポートフォリオが選択されていないプロジェクトは、「値なし」という名前のグループに表示されます。

その結果、複数のグループにオブジェクトを表示することはできません。 また、リストにオブジェクトが表示されるかどうかが、フィルターによって完全に制御されます（レポートを実行しているユーザーがそのオブジェクトを表示する権限を持っている場合）。

**質問**

ベロシティを追跡するために他のレポートをお勧めしますか？ 詳細を説明するのに十分な時間がないことがわかっているので、上位レベルのレコメンデーションだけでも素晴らしいです。

**回答**

他のレポートと同様に、最初に行う必要があるのは、知りたいことを決定することです。 次のステップは、その情報にアクセスすることです。場合によっては、その情報の追跡を開始する必要があります。

私が実際の期間と2種類の予定期間を比較することを決めた理由の1つは、それが速度に関する興味深い洞察を提供すると思ったからです。 データも既に利用可能だったので、トラッキングを開始する必要はありませんでした。 いくつか計算すれば、それについて報告できる形式でデータを抽出できる。

あるいは、タスクやプロジェクトに関する他の情報を追跡することもできます。

Workfrontにはベロシティに関するレポートが組み込まれていません。そのため、ベロシティを判断するために知っておくべきことをブレインストーミングし、追跡すべき点を確認することをお勧めします。

**質問**

カラム単位で何か計算できることはありますか？ 計算フィールドをカスタムフォームから呼び出す代わりに？

**回答**

テキストモードで値式を使用して、これらの計算を行うことができたでしょう。 最初の期間や最初のコミット日を行うことはできませんでしたが、変更されない場所でキャプチャする必要がありました。

**質問**

**質問**

例えば先月など、定義された期間のステータスの変更を検索するレポートの作成方法。

**回答**

Workfrontの分析機能では、ステータスの変更など、過去のデータを容易に表示できます。

メモレポートを使用してステータス変更情報を取得することもできます。 「プロジェクトのステータス」フィールドをトラッキングしている場合は、プロジェクトのステータス変更をフィルタリングして表示できます。

まず、設定/インターフェイス/フィードの更新に移動し、プロジェクトステータスがトラッキング対象のビルトインフィールドの1つであることを確認します。 そうでない場合は追加する必要があります。

次に、メモレポートを作成し、次の操作を行います。

「列（表示）」タブで、次の操作を行います。

* 「メモテキスト」列を「監査テキスト」に置き換えます。 これにより、ステータスがおよびから変更された内容に関する情報が表示されます
* 「プロジェクト：名前」列と「エントリ日」列は残します
* 「入力日」列をクリックし、列設定パネルの「この列で並べ替え」にチェックを入れます。 最新のステータスの変更を一番上に表示したい場合は、降順に並べ替えます。

「グループ化」タブで、次の操作を行います。

* プロジェクト別グループ化：名前

「フィルター」タブで、次のフィルタールールを作成します。

* メモ >>監査タイプ >>次と等しい>> ステータスの変更
* メモ入力日でフィルタリングする追加ルールを追加します。 フィルターからこれらを除外し、代わりにレポートプロンプトを使用することをお勧めします
* 必要に応じて、プロジェクト、ポートフォリオ、その他のデータでフィルタリングできます。

**質問**

プランナーとして、他のユーザーのレポートを取得できますか？

**回答**

プランナーは、レポートを作成し、ユーザー以外のユーザーでも、任意のユーザーと共有できます。 レポートを表示する際は、「レポートアクション/共有」に移動し、「レポートアクセス」ボックスの右上にある歯車アイコンをクリックします。 「外部ユーザーに公開」を選択します。 オプション： 提供されたリンクをコピーして、誰にでも送信できます。 レポートがブラウザーでリアルタイムに表示されます。
